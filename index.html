MOD 2 Reversing Malicious Code — x86, Assembly, Windows API
2M 5M 10M ✓
2 MARK DEFINITIONS
Microcode vs Machine CodeMicrocode = low-level CPU internal instructions controlling hardware. Machine code = binary instructions CPU executes directly. Microcode sits below machine code in abstraction.
Low-Level LanguagesClose to hardware, minimal abstraction, eg. Assembly. Fast execution but hardware-dependent and hard to write.
High-Level LanguagesHuman-readable, portable, abstracted from hardware. eg. Python, C. Easier to write but slower than low-level.
Interpreted LanguagesExecuted line-by-line by interpreter at runtime. eg. Python, JS. Flexible but slower than compiled.
Disassembling + Control FlowDisassembler converts binary → assembly. Control flow identified via JMP/JNE/JE for conditionals, backward jumps for loops, CALL for functions.
KeyloggingRecords keystrokes via API hooks (SetWindowsHookEx). Detected by monitoring suspicious API calls during dynamic analysis.
Data ExfiltrationUnauthorized extraction of data using HTTP POST, DNS tunneling, or encrypted C2 channels.
5 MARK SKELETONS
Q32 — x86 Registers
  • General purpose: EAX (accumulator), EBX (base), ECX (counter), EDX (data)
  • ESP = stack pointer, EBP = base pointer — critical for stack frame analysis
  • EIP = instruction pointer, always points to next instruction to execute
  • Flags: ZF (zero), CF (carry), SF (sign) — control all conditional jumps
  • Malware manipulates EIP to hijack execution flow
Q35 — Registry Manipulation
  • Malware uses HKCU\...\CurrentVersion\Run for persistence across reboots
  • Stores C2 config, encryption keys in registry values
  • Deletes keys post-execution to erase tracks
  • Detected via RegCreateKey, RegSetValueEx API monitoring
  • Example: Zeus, most RATs and ransomware use registry for persistence
Q36 — Trace Execution Flow
  • Analyst follows conditional branches and function calls using disassembler
  • Control Flow Graph (CFG) visualizes all possible execution paths
  • Key instructions: JMP, CALL, RET, CMP, JNE/JE
  • Debugger used to step through live execution and observe register values
  • Goal: find decision point where malicious action triggers
Q31 — Levels of Abstraction
  • Hardware → Microarchitecture → Machine Code → Assembly → High-Level → Application
  • Analysts work at assembly and machine code level
  • Higher levels hide malicious intent, lower levels reveal actual behavior
  • RE works bottom-up: binary → assembly → logic understanding
  • Malware authors use high-level code, analysts must reverse to low-level
10 MARK SKELETONS — PICK ONE IN EXAM
★ Q42 — Heap and Stack (MOST LIKELY)
WHAT
Stack = LIFO memory for function calls, local vars, return addresses. Heap = dynamic memory allocated at runtime via malloc/VirtualAlloc.
HOW
Stack: function called → frame created → EBP saved → locals pushed → RET destroys frame. Heap: malware calls VirtualAlloc → writes shellcode → executes from heap.
WHY IT MATTERS
Buffer overflow targets stack to overwrite return address and redirect EIP. Heap spraying fills heap with shellcode to increase exploit hit rate. Both = code execution.
EXAMPLE
Classic stack overflow exploits (CVE patterns), heap spray attacks in IE browser exploits. Cobalt Strike uses heap injection to hide shellcode.
★ Q40 — Windows API Level Analysis (VERY LIKELY)
WHAT
Malware communicates with OS through Windows API. Analysts identify behavior by spotting suspicious API call patterns during static/dynamic analysis.
HOW — KEY API CHAINS
File ops: CreateFile/WriteFile/DeleteFile. Registry: RegCreateKey/RegSetValue. Injection: OpenProcess → VirtualAllocEx → WriteProcessMemory → CreateRemoteThread. Network: InternetOpen/HttpSendRequest. Anti-debug: IsDebuggerPresent.
WHY
Each API chain reveals capability: injection chain = process injection, network APIs = C2 comms, file APIs = ransomware encryption, registry APIs = persistence. No need to understand full code.
EXAMPLE
Zeus banking trojan identified entirely through API call pattern. WannaCry's file encryption chain visible via CreateFile+CryptEncrypt API sequence.
★ Q38 — Assembly Logic Structures
WHAT
Disassembler converts binary to assembly. Analyst identifies high-level logic constructs like if-else, loops, switch cases, and function calls from raw assembly instructions.
HOW — EACH CONSTRUCT
If-else: CMP + JE/JNE. Loops: backward JMP, ECX counter, LOOP instruction. Switch: jump tables (array of addresses). Functions: CALL pushes return addr, RET pops it. Frame: PUSH EBP / MOV EBP,ESP prologue → POP EBP / RET epilogue.
WHY
Recognizing constructs lets analyst understand malware decision logic without source code. Find anti-debug checks, decryption loops, C2 connection attempts hidden in binary.
EXAMPLE
Anti-debug loop = ECX initialized to 1000, backward JMP creates loop checking IsDebuggerPresent 1000 times before payload runs.
MOD 3 Malicious Web & Document Files — PDF, RTF, JS, Office

2 MARK DEFINITIONS
Scanning PDFs for MalwareAutomated tools check embedded JS, exploit payloads, suspicious streams using signature detection + sandboxing.
RTF FormatRich Text Format — uses control words and groups, can embed OLE objects, commonly abused via malformed control words to deliver exploits.
SRPSoftware Restriction Policies — Windows mechanism to block unauthorized program execution based on hash, path, or certificate rules.
Code MinifiersTools that remove whitespace/comments to reduce file size and improve performance. NOT obfuscation — doesn't change logic, just compresses.
Control Words in RTFBackslash keywords controlling formatting eg. \rtf1. Can be abused to embed malicious OLE objects or trigger exploits.
OOXMLOffice Open XML — MS Office format that is a ZIP containing XML files. Macros stored in vbaProject.bin. Can hide malicious macros inside.
Obfuscation PurposeHides actual code logic from analysts and AV tools by making it unreadable while keeping full functionality intact.

Q52 — JS Obfuscation Techniques
  • String encoding: strings converted to hex/unicode to hide C2 URLs
  • Variable renaming: meaningful names replaced with random characters
  • Control flow flattening: normal flow replaced with complex switch statements
  • Dead code insertion: fake irrelevant code added to confuse analyst
  • Dynamic evaluation: eval() used to execute encoded strings at runtime
  • Target: hide C2 URLs, malicious API calls, payload delivery logic
Q56 — How PDF Delivers Malware
  • Embedded JavaScript executes on open via PDF reader vulnerability
  • Malicious hyperlinks redirect to exploit kit landing pages
  • Exploit payloads target buffer overflows in PDF reader (Adobe, Foxit)
  • Social engineering tricks user into opening attachment or enabling content
  • Streams contain obfuscated shellcode decoded at runtime
  • Example: CVE-2010-0188 Adobe Reader exploit, Zbot delivered via PDF
MOD 4 In-Depth Malware Analysis — Packing, Fileless, Injection, Memory
2M 5M 10M ✓
2 MARK DEFINITIONS
Packed MalwareMalware compressed/encrypted to hide code and evade AV signature detection. Unpacks itself at runtime in memory.
Elements of Packed MalwarePacker tool + encrypted payload + stub (decompression routine) + execution transfer mechanism.
Why PackingBypass signature AV, complicate reverse engineering, hide strings/imports, reduce detectable footprint.
Packed vs Normal MalwarePacked = high entropy sections, few readable strings, abnormal PE entry point, obfuscated imports. Normal = readable strings, standard structure.
Packing vs UnpackingPacking = attacker hides code. Unpacking = analyst reveals hidden payload via dynamic debugging or memory dumping.
Detect Packed MalwareEntropy analysis, PE header inspection, import table anomalies, signature detection, behavioral monitoring.
Analyze Packed MalwareStatic analysis → dynamic debugging → memory dump at OEP (Original Entry Point) → unpacking tools (UPX, OllyDump).
5 MARK SKELETONS
Q8 — How Packed Malware Looks
  • High entropy sections (close to 8.0) due to encryption/compression
  • Abnormal file size — smaller than expected for apparent functionality
  • Suspicious entry point — jumps to non-standard section
  • Very few readable strings — most content is encrypted/compressed
  • Obfuscated or missing import table — hides API calls from static analysis
Q12 — Code Injection
  • Attacker injects malicious code into a legitimate running process
  • Injected code executes with the privileges of the host process
  • Types: DLL injection, shellcode injection, reflective loading
  • Evades process-based AV since malicious code runs under trusted PID
  • Detected via memory scanning for injected regions with RWX permissions
Q14 — On-Disk vs In-Memory Attacks
Factor On-Disk In-Memory
Storage Written to filesystem RAM only, no file
Detection AV can scan files Harder, needs memory scan
Persistence Survives reboot Lost on reboot
Forensics File artifacts remain Volatile, hard to recover
10 MARK SKELETONS
★ Q15 — What is Packing and Why (MOST LIKELY)
WHAT
Packing = compressing or encrypting a malware executable using a packer tool to produce a new binary that unpacks itself at runtime. The visible binary hides the real payload.
HOW IT WORKS
Packer wraps original payload → adds stub (decompression routine) as new entry point → stub decrypts/decompresses payload into memory at runtime → transfers execution to Original Entry Point (OEP).
WHY ATTACKERS USE IT
Bypass signature-based AV (signature of original malware not visible), slow down analyst (must unpack before analysis), hide readable strings/API imports, create unique variants easily.
EXAMPLE + DETECTION
UPX is most common packer. Emotet used custom packers. Detection: entropy check (high entropy = packed), PE section analysis, import table inspection, OllyDbg to find OEP via memory dump.
★ Q17 — Fileless Attack + How It Works (VERY LIKELY)
WHAT
Fileless malware never writes to disk. Lives entirely in memory and uses legitimate system tools (LOLbins) to execute. No file = no AV signature to detect.
STAGES
1. Initial compromise (phishing/exploit) → 2. Script execution (PowerShell/WMI/macros) → 3. Memory injection (shellcode into legit process) → 4. Persistence (registry-resident/scheduled tasks) → 5. Data exfiltration via encrypted channel.
WHY DANGEROUS
No file artifact means traditional AV signature matching fails. Uses trusted Windows tools so behavior-based detection also struggles. Volatile nature makes forensics hard. Evades most endpoint security.
EXAMPLE
Cobalt Strike — beacon injected into memory via PowerShell. PowerShell Empire — entirely memory resident. WMI-based persistence survives reboot without any disk file.
★ Q19 — Detect Fileless + Memory Forensics
DETECT FILELESS
Behavioral EDR solutions monitor process behavior not just files. PowerShell Script Block Logging captures all PS commands. Memory scanning for anomalous RWX regions. AMSI (Antimalware Scan Interface) intercepts script execution.
MEMORY FORENSICS — HOW
Acquire RAM image (live acquisition or hibernation file) → load into Volatility → extract running processes, network connections, injected code, registry hives, decrypted strings all from RAM snapshot.
WHY RAM FORENSICS WORKS
Even fileless malware must exist in RAM to execute. Memory contains decrypted payload, C2 URLs, injected shellcode — everything attacker tried to hide on disk is visible in RAM.
EXAMPLE
Volatility tool used to extract Cobalt Strike beacon config from memory. malfind plugin detects injected memory regions. netscan reveals hidden C2 connections not visible in process list.
MOD 5 Self-Defending Malware — Evasion, Anti-Analysis, Process Hollowing
2M 5M 10M ✓
2 MARK DEFINITIONS
Malware Self-DefenseTechniques malware uses to resist analysis: anti-debugging, anti-VM detection, code obfuscation, encryption.
Self-Defending NetworkNetwork that automatically detects, prevents, and responds to threats using integrated security mechanisms without human intervention.
Process HollowingCreates a legitimate process in suspended state → hollows out its memory → writes malicious code → resumes under original PID. Malware impersonates legit process.
TLS CallbacksThread Local Storage callbacks execute BEFORE the main entry point. Malware uses these to run anti-debug checks before analyst's debugger can attach.
Controlled EnvironmentIsolated VM or sandbox used to safely execute and observe malware behavior without risk to production systems.
Packer Behavior AnalysisObserve unpacking routine, entry point redirection, memory allocation patterns, and runtime decryption by running packed sample in debugger.
Sandbox Evasion TechniquesVM detection, timing attacks, environment checks (user files, screen resolution), user interaction checks, anti-debugging methods.
5 MARK SKELETONS
Q33 — Process Hollowing + Examples
  • CreateProcess with SUSPENDED flag → legit process created but not running
  • NtUnmapViewOfSection hollows out the process memory image
  • VirtualAllocEx + WriteProcessMemory writes malicious code into hollow
  • SetThreadContext redirects entry point, ResumeThread resumes execution
  • Result: malicious code runs under trusted PID (eg. svchost.exe)
  • Examples: Dridex, TrickBot both impersonated svchost.exe this way
Q34 — Sandboxes + Sandbox-Evading Malware
  • Sandbox = isolated environment to detonate and observe malware safely
  • Malware checks for VM artifacts: registry keys, MAC addresses, screen resolution
  • Timing attacks: Sleep() for longer than sandbox timeout, then execute
  • No user interaction = sandbox (no mouse movement, no recent documents)
  • If sandbox detected → malware stays dormant, shows no malicious behavior
  • Example: Emotet checked for virtual disk names before executing
Q29 — How Self-Defending Networks Work
  • Continuous real-time monitoring of all network traffic and endpoints
  • Automated threat intelligence feed updates detection rules dynamically
  • Anomaly detection flags deviations from baseline behavior
  • Automated response: isolate infected host, block malicious IP, alert team
  • Adaptive — learns from new threats and updates its own defenses
10 MARK SKELETONS
★ Q36 — Three Categories of Malware Self-Defense (MOST LIKELY)
WHAT
Malware uses three layers of self-defense: Anti-Analysis (defeat static tools), Anti-Debugging (defeat dynamic analysis), Anti-Virtualization (defeat sandbox analysis). Each targets a different analysis phase.
HOW EACH WORKS
Anti-analysis: obfuscation, encryption, packing hides code from static scanners. Anti-debugging: IsDebuggerPresent API, timing checks (RDTSC), SEH manipulation, TLS callbacks run before debugger attaches. Anti-VM: checks VM registry keys, CPUID instruction, MAC address, screen resolution, disk size.
WHY IT MATTERS
Each layer forces analyst to spend more time and resources. Combined they make automated analysis fail completely. Forces manual expert analysis which is expensive and slow. Buys attacker time for attack to succeed.
EXAMPLE
Emotet used all three layers. Zeus used anti-debug + encryption. Modern APT malware (Cobalt Strike) uses anti-VM as first check. Conclusion: layered self-defense is standard in sophisticated malware today.
★ Q39 — Evasion Techniques in Detail (VERY LIKELY)
WHAT
Malware evasion = techniques to avoid detection by security tools. Operates at signature level (fool AV), behavior level (look legitimate), and environment level (detect analysis tools).
HOW — KEY TECHNIQUES
Polymorphism: changes code signature every infection cycle, same function but different bytes. Metamorphism: rewrites entire code logic, hardest to detect. Encryption: payload encrypted, only decrypts in memory. Packing: compresses binary. Behavioral obfuscation: mimics legitimate process network/file behavior.
WHY EACH BEATS SPECIFIC DEFENSE
Polymorphism beats signature AV. Metamorphism beats heuristic AV. Encryption beats string-based detection. Behavioral obfuscation beats behavior-based EDR. No single defense catches all — defense in depth required.
EXAMPLE
WannaCry had polymorphic variants in the wild. Virut was metamorphic. Most modern ransomware encrypts its payload. Conclusion: attackers layer evasion techniques to defeat multiple detection layers simultaneously.
★ Q40 — Process Hollowing Risks + Protection
WHAT
Process hollowing = attacker creates a legitimate suspended process, replaces its memory with malicious code, resumes it. Malicious code runs under trusted process identity (PID, name, path).
HOW THE ATTACK WORKS
CreateProcess (SUSPENDED) → NtUnmapViewOfSection (hollows memory) → VirtualAllocEx (allocates space) → WriteProcessMemory (writes payload) → SetThreadContext (new entry point) → ResumeThread (executes).
RISKS
Stealth execution — malicious code appears as svchost.exe or explorer.exe in process list. Privilege abuse — inherits privileges of hollowed process. Bypasses process-based AV whitelisting completely.
PROTECTION + EXAMPLE
EDR tools monitor CreateProcess+NtUnmapViewOfSection combination as IoC. Memory integrity checks detect mismatches between on-disk and in-memory process image. Example: Dridex banking trojan, TrickBot both used process hollowing against svchost.exe.
